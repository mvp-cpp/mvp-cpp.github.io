Section 3

Running HIMVP

This section lists common HIMVP workflows. Use one analysis mode per command unless the workflow explicitly supports automatic clustering inside GLM or MLM.

Quick Start

Start with a small and explicit command before launching a full analysis. The examples below assume that genotype files use the prefix data/geno, the phenotype file is data/pheno.txt, and output files will be written under result/. 

# GLM GWAS
himvp --glm --bfile data/geno --pheno data/pheno.txt --out result/glm

# MLM GWAS
himvp --mlm --bfile data/geno --pheno data/pheno.txt --out result/mlm

# PCA
himvp --pca --bfile data/geno --npc 5 --out result/pca

# GRM construction
himvp --make-grm --bfile data/geno --out result/grm
  • Use --glm first to verify basic phenotype, genotype and covariate formatting.
  • Use --mlm when relatedness or population structure needs stronger control.
  • Use --pca or --make-grm separately when you want to inspect intermediate files.
  • Use a fresh --out prefix for each run to avoid overwriting previous output.

Output Prefix and Threads

Use --out to set the output prefix and --threads to set the number of CPU threads. This rule applies across GLM, MLM, PCA, GRM, clustering, meta-analysis and plotting workflows.

OptionPurposeRecommendation
--outPrefix for result tables, figures and logs.Use a directory plus a descriptive prefix, for example result/trait1_glm.
--threadsNumber of CPU threads.Use a value that leaves resources for the operating system and other jobs.
--stepNumber of markers loaded into RAM at a time.Increase for speed when memory is sufficient; decrease when memory is tight.

Output prefix rule. HIMVP appends suffixes such as .assoc.txt or plot file extensions to the prefix. The prefix itself should not include those suffixes.

GLM Analysis

Use --glm as the first GWAS scan to verify phenotype, genotype, covariate and association-output formatting.

Requires: phenotype file and one genotype prefix. See Phenotype File and Genotype Files.

himvp --glm \
    --bfile demo \
    --pheno demo.phe \
    --pheno-pos 2 \
    --dcovar 4,5 \
    --qcovar 6 \
    --maf 0.01 \
    --out result/glm_demo
  • Output: association table under the selected --out prefix.
  • Use --add for additive genetic effect or --dom for dominant effect. If neither is supplied, additive effect is used.
  • Add covariates only after confirming their column positions and data types in the input-format section.

MLM Analysis

Use --mlm when population structure or kinship may inflate GLM statistics. If no --grm is provided, HIMVP can construct the relationship matrix from genotype data.

Requires: phenotype file, genotype prefix and optionally a GRM prefix. See Relationship Matrix File.

himvp --mlm \
    --bfile demo \
    --pheno demo.phe \
    --pheno-pos 2 \
    --npc 3 \
    --out result/mlm_demo
  • Output: association table and model logs under the selected --out prefix.
  • --npc adds principal components to the fixed-effect part of the GWAS model.
  • --max-iter controls the maximum number of variance component iterations.
  • --mlm-path mkm uses marker-by-marker variance handling for full MLM scans; --mlm-path emmax estimates variance once and reuses it for faster approximate scans.
  • For production MLM runs, keep the GRM prefix, version and command line together with the output.

Multi-trait Analysis

Pass multiple phenotype columns to analyze several traits in one command.

Requires: one phenotype file whose selected trait columns share the same individual ID column.

himvp --mlm \
    --bfile demo \
    --pheno demo.phe \
    --pheno-pos 2,3,4 \
    --out result/mlm_multi_trait
  • Output: trait-specific columns in the association result table.
  • Each selected trait may have missing values encoded as NA.
  • If --vg and --ve are supplied, provide one value per selected trait in the same order as --pheno-pos.

PCA

Use --pca with genotype data or a pre-computed GRM to inspect population structure.

Requires: genotype prefix or GRM prefix, plus the requested number of PCs.

himvp --pca --bfile demo --npc 5 --out result/pca
himvp --pca --grm demo_grm --npc 5 --out result/pca_from_grm
  • Output: PC scores and variance summary files.
  • Use PCA output for structure review, plotting and clustering input.

Model correction. For GLM or MLM, specify --npc in the association command rather than passing a PC file with --pc.

GRM Construction

Use --make-grm to construct reusable genomic relationship matrix files.

Requires: genotype prefix. The resulting GRM can be reused by MLM, PCA and clustering workflows.

himvp --make-grm \
    --bfile demo \
    --grm-method 1 \
    --out result/demo_grm
  • Output: GRM binary matrix and ID files under the selected prefix.
  • --grm-method selects the relationship-matrix construction method.
  • Use --bitwise-grm for BED-based raw additive GRM construction when that optimized path is appropriate.
  • Use --float-prec if you need lower memory use for matrix-heavy steps.

Population Clustering

Use --cluster or --auto-cluster for HDBSCAN-based clustering from genotype, GRM or PC input.

Requires: one structure input source. See Principal Component File and Relationship Matrix File.

himvp --cluster --bfile demo --npc-cluster 3 --out result/cluster
himvp --cluster --pc demo.pc --npc 3 --out result/cluster_from_pc
  • Output: cluster labels and optional parameter/embedding files.
  • Use clustering when known population labels are unavailable or need cross-checking.
  • When clustering from genotype data, HIMVP first derives the requested number of PCs for clustering.
  • Use --npc-cluster for the number of PCs used in clustering from genotype or GRM input.
  • --use-umap can add UMAP before clustering; tune --umap-nn and --umap-md only after the basic command works.
  • --use-kdtree can reduce memory cost for clustering neighborhood queries.
himvp --auto-cluster \
    --bfile demo \
    --npc-cluster 10 \
    --use-umap \
    --umap-nn 15 \
    --umap-md 0.1 \
    --out result/cluster_umap

Parameter compatibility. Do not combine --grid with --use-umap. Grid optimization is for the automatic clustering workflow without UMAP.

GPU Acceleration

GPU options are available only when HIMVP is built with CUDA support. Use GPU acceleration mainly for computationally intensive MLM workflows and benchmark CPU versus GPU results on a representative subset before running a full dataset.

Requires: CUDA-enabled HIMVP build and a working CUDA runtime.

himvp --mlm \
    --bfile demo \
    --pheno demo.phe \
    --device cuda \
    --gpus 0,1 \
    --evd-method mg \
    --out result/mlm_gpu
OptionValuesUse
--deviceauto, cpu, cudaSelects compute device.
--gpusComma-separated IDsSelects GPU devices, for example 0,1.
--evd-methodauto, dc, mg, cs, bk, jacobi, qr, r3, svd_zControls eigendecomposition path accepted by current runtime validation.
--mlm-varauto, cpu, gpuSelects whether MLM variance estimation runs automatically, on CPU or on GPU.
--mlm-var-methodai, he, hiControls variance estimation strategy.
--mlm-pathmkm, emmaxSelects MLM algorithmic path.

Runtime validation. The current source tree still has a few differences between CLI help text and later parameter validation. This table lists values that pass the current runtime checks.

SettingPractical meaningWhen to use it
--mlm-var-method aiAI-REML variance estimation.Default choice for standard MLM runs.
--mlm-var-method heHE regression without REML iteration.Fast exploratory runs or rough variance estimates.
--mlm-var-method hiHE initialization followed by AI-REML refinement.Use when HE initialization may help convergence.
--mlm-var autoLet HIMVP choose the variance-estimation placement.Recommended unless benchmarking CPU/GPU behavior.
--mlm-var cpuRun variance estimation on CPU.Conservative choice for reproducible production runs.
--mlm-var gpuRequest GPU variance estimation.Experimental in the current source tree; verify logs before relying on it.

Benchmark Mode

Use benchmark mode to write phase-level timing information. This is useful when comparing CPU and GPU settings, tuning --step, or preparing a performance report for a release.

Requires: any valid analysis command plus benchmark output options.

himvp --mlm \
    --bfile demo \
    --pheno demo.phe \
    --benchmark \
    --benchmark-out result/mlm_benchmark \
    --out result/mlm

HIMVP writes <benchmark-out>.timing.json. Keep this JSON file with the association output so runtime comparisons can be reproduced later.

  • Run benchmark commands on an otherwise quiet machine when possible.
  • Use the same input files and model options when comparing two devices.
  • Record CPU model, GPU model, memory size and HIMVP version together with the timing file.

Plot Generation During Analysis

When the plotting module is enabled, add --plot to generate figures after analysis. This is convenient for routine GWAS runs because association tables and plots are generated from the same command and output prefix.

Requires: a plotting-enabled HIMVP build and an analysis that produces plot-ready output.

himvp --glm \
    --bfile demo \
    --pheno demo.phe \
    --plot \
    --plot-theme nature \
    --out result/glm_plot
  • Output: association table plus figures under the selected prefix.
  • Use --plot-only from an existing result file when you only need to regenerate figures.
